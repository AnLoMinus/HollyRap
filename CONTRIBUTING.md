## 💡 איך אני יכול לתרום?

### 📝 דיווח על באגים

לפני יצירת issue לדיווח על באג:

1. בדוק את [Issues הקיימים](https://github.com/AnLoMinus/hollyrap/issues)
2. ודא שהבאג לא דווח כבר
3. אם הבאג לא קיים, צור issue חדש

**מידע חשוב לכלול:**
- תיאור ברור של הבאג
- צעדים לשחזור
- התנהגות צפויה vs התנהגות בפועל
- גרסה/סביבה רלוונטית

### ✨ הצעת תכונות חדשות

אם יש לך רעיון לתכונה חדשה:

1. בדוק את [Issues הקיימים](https://github.com/AnLoMinus/hollyrap/issues)
2. ודא שהרעיון לא הוצע כבר
3. צור issue חדש עם תווית "enhancement"

**מידע חשוב לכלול:**
- תיאור מפורט של התכונה
- מקרה שימוש ברור
- יתרונות וישימות

### 📚 שיפור תיעוד

תיעוד טוב הוא קריטי! אתה יכול לעזור:

- תיקון שגיאות הקלדה
- הוספת דוגמאות
- שיפור בהירות
- הוספת סעיפים חסרים

### 🎤 הוספת באטלים חדשים

אנחנו תמיד מחפשים באטלים חדשים!

**קריטריונים לבאטל טוב:**
- מסר חיובי וערכי
- עומק תורני וחוכמה
- חריזה וזרימה טובה
- מבנה ברור וקוהרנטי

**תהליך:**
1. צור קובץ markdown חדש ב-`content/battles/`
2. עקוב אחר הפורמט של הבאטלים הקיימים
3. הוסף metadata מתאים
4. פתח Pull Request

### 🐛 תיקון באגים

1. בדוק את [Issues הפתוחים](https://github.com/AnLoMinus/hollyrap/issues)
2. בחר באג שאתה יכול לטפל בו
3. צור fork של המאגר
4. עשה את התיקון
5. פתח Pull Request

## 🛠️ התפתחות

### הכנת סביבת פיתוח

1. **Fork** את המאגר
2. **Clone** את ה-fork שלך:
   ```bash
   git clone https://github.com/YOUR_USERNAME/hollyrap.git
   ```
3. **צור branch** חדש:
   ```bash
   git checkout -b feature/amazing-feature
   ```

### תהליך עבודה

1. עשה שינויים בקוד/תוכן
2. בדוק שהכל עובד
3. commit השינויים שלך:
   ```bash
   git commit -m 'Add amazing feature'
   ```
4. push ל-branch:
   ```bash
   git push origin feature/amazing-feature
   ```

## 🔄 תהליך Pull Request

1. ודא שה-branch שלך מעודכן עם `main`:
   ```bash
   git checkout main
   git pull upstream main
   git checkout feature/amazing-feature
   git rebase main
   ```

2. פתח Pull Request:
   - עבור ל-GitHub repository שלך
   - לחץ על "New Pull Request"
   - בחר את ה-branch שלך
   - מלא את התבנית

3. המתן לבחינה:
   - נבדוק את ה-PR בהקדם האפשרי
   - ייתכן שנבקש שינויים
   - לאחר אישור, ה-PR ימוזג

## ✅ קריטריונים לבחינה

**כל Pull Request צריך:**

- ✅ להיות עקבי עם סגנון הקוד הקיים
- ✅ לכלול תיעוד מתאים
- ✅ להיות מובן וברור
- ✅ לעבור בדיקות בסיסיות
- ✅ לעמוד בקוד ההתנהגות

**לתוכן (באטלים):**

- ✅ מסר חיובי וערכי
- ✅ עומק תורני וחוכמה
- ✅ חריזה וזרימה טובה
- ✅ פורמט נכון (Markdown)
- ✅ metadata מלא

## 📝 עקרונות כתיבה

### באטלים

- **מבנה:** פתיחה → התפתחות → שיא → סיום
- **חריזה:** חשוב אבל לא על חשבון המסר
- **אורך:** 12-16 שורות לכל צד (או לפי נושא)
- **תוכן:** מסר חיובי, עומק תורני, חוכמה מעשית

### תיעוד

- כתוב בצורה ברורה ופשוטה
- הוסף דוגמאות כשיש צורך
- שמור על עקביות בפורמט
